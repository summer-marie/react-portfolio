# Site Architecture

## Objective

Create a clear, focused portfolio structure that helps hiring managers quickly understand:

* Who I am
* What kind of engineer I am
* What I have built
* How I think
* How to contact me

The site should feel intentional and easy to explore without overwhelming visitors.

---

# Primary Navigation

Use a simple top navigation:

* Home
* Work
* About
* Resume
* Contact

Optional utility controls:

* Theme toggle
* GitHub link
* LinkedIn link

Keep the navigation compact and visible without dominating the page.

---

# Page Structure

## 1. Home

The homepage should communicate the strongest version of my story in under 30 seconds.

### Sections

1. Hero
2. Positioning statement
3. Selected work
4. Engineering approach
5. Current focus
6. Contact call to action
7. Footer

### Hero Goals

The hero should immediately communicate:

* I am a forward deployed engineer
* I build useful products
* I work across product, engineering, and users
* I am comfortable solving ambiguous problems

### Hero Content

Include:

* Short role label
* Strong headline
* Supporting sentence
* Primary CTA: View Work
* Secondary CTA: Contact Me
* Small availability or current-focus indicator

Avoid:

* Large headshot
* Long biography
* Skill badges
* Technology logo walls

---

## 2. Work

The Work page should present a curated set of projects and client work.

Do not organize projects around whether they are technically finished.

Organize them around whether they provide strong evidence of:

* Product thinking
* Technical ownership
* Real-world impact
* User collaboration
* Interesting engineering decisions

### Project Categories

Possible categories:

* Featured Products
* Client Work
* Experiments
* In Progress

Only the strongest work should appear prominently.

### Project Card Content

Each card may include:

* Project name
* Short problem statement
* My role
* Status
* Selected technologies
* One strong visual
* Link to case study or live product

---

## 3. Project Case Study

Each featured project should use the same case-study structure.

### Case Study Sections

1. Project overview
2. Problem
3. Users or stakeholders
4. Constraints
5. My role
6. Approach
7. Key engineering decisions
8. Product or technical outcomes
9. Lessons learned
10. Links

### Optional Visuals

* Product screenshots
* System diagrams
* Architecture diagrams
* Before-and-after comparisons
* Workflow illustrations
* Short animated demonstrations

The case study should explain judgment and decision-making, not only implementation details.

---

## 4. About

The About page should feel personal without becoming a full autobiography.

### Sections

1. Short introduction
2. How I approach engineering
3. What forward deployment means to me
4. Current interests
5. Selected experience
6. Personal details or interests
7. Contact CTA

### Tone

* Direct
* Thoughtful
* Warm
* Confident
* Not overly formal

Keep the page concise and specific.

---

## 5. Resume

The Resume page should be easy to scan.

### Include

* Professional summary
* Experience
* Selected projects
* Education
* Technical strengths
* Download PDF button

The HTML version should be the primary experience.

The PDF should be available as a secondary download.

---

## 6. Contact

The Contact page should make starting a conversation easy.

### Include

* Short invitation
* Email address
* LinkedIn
* GitHub
* Optional contact form
* Current availability

Avoid long forms or unnecessary required fields.

---

# Homepage Hierarchy

The homepage should follow this order:

## 1. Hero

Immediate positioning and CTA.

## 2. Selected Work

Show the strongest evidence early.

## 3. How I Work

Explain the qualities that make me a forward deployed engineer.

Possible themes:

* Understand the user
* Clarify the problem
* Build quickly
* Learn from feedback
* Improve the system

## 4. Current Focus

Show what I am currently learning, building, or exploring.

## 5. Contact CTA

End with a direct invitation to talk.

---

# Content Priority

When deciding what appears on the site, use this order:

1. Evidence of ownership
2. Product impact
3. Engineering decisions
4. Collaboration
5. Technical depth
6. Technology names

Technologies should support the story, not become the story.

---

# Responsive Behavior

## Desktop

* Spacious layout
* Layered visuals
* Ambient gradients
* Subtle animated elements
* Multi-column project sections where appropriate

## Tablet

* Reduced decorative elements
* Two-column layouts become stacked where needed
* Preserve spacing and hierarchy

## Mobile

* Single-column layout
* Simplified animation
* Large tap targets
* Reduced ambient effects
* Clear CTAs
* No horizontal scrolling

---

# Navigation Behavior

* Sticky or semi-sticky header
* Transparent or lightly tinted initial state
* Stronger surface and border after scrolling
* Active page indicator
* Smooth transitions between pages
* Mobile menu should open quickly and remain simple

---

# Footer

Include:

* Name
* Short positioning statement
* Email
* GitHub
* LinkedIn
* Copyright
* Optional “Built with” note

Keep the footer useful and visually quiet.

---

# Version 1 Scope

Build only:

* Home
* Work
* About
* Resume
* Contact
* Reusable project detail template

Do not build yet:

* Blog
* Search
* CMS
* Complex filtering
* Separate pages for every experiment
* Interactive resume timeline
* Large custom illustration system

These can be added after the core portfolio is complete.
